Case 01

자동차선 적부 최적화 Backend Service

차량용 운송 선박의 적부 계획 최적화 모델을 기존 업무 시스템에서 사용할 수 있도록, 선박 공간 데이터 모델, 장시간 추론 API, scale-out 상태 관리, 표준 interface를 설계했습니다.

FastAPIPydanticQueueRedis/MongoDBKubernetes

문제 정의

  • 선박은 여러 층과 연결 통로로 구성되고, 각 층마다 적재 가능 공간과 연결 관계가 다릅니다.
  • 차량은 높이, 선적지, 양하지, 용량이 다르고, 내리는 순서를 고려하지 않으면 불필요한 이동이 발생합니다.
  • 현업 도면의 다각형 공간을 그대로 모델에 넣기 어렵기 때문에 계산 가능한 추상화가 필요했습니다.

왜 비동기 구조인가?

  • 입력 난도에 따라 모델 추론 시간이 5~20분으로 달라져 동기 API timeout을 예측하기 어렵습니다.
  • Polling은 서버 자원을 낭비하고, 엔진 서버는 고비용 리소스라 요청 진입량을 제한해야 했습니다.
  • API 서버는 검증·전처리·후처리·callback에 집중하고, engine 서버는 표준 입력을 받아 추론만 수행하도록 분리했습니다.

Scale-out에서 생긴 문제

API pod와 engine pod를 여러 개 띄우면 요청을 받은 pod와 결과가 돌아오는 pod가 달라질 수 있습니다. 그래서 pod local memory에 요청 상태를 두면 race-condition에 가까운 문제가 생깁니다.

Decision요청 상태와 결과를 공용 queue/store로 분리하고, pod는 stateless하게 동작하도록 설계했습니다.

Case 02

Personal AI Agent Runtime Platform

개인별 Agent를 사용할 수 있도록 사용자 runtime을 격리하고, 공용 skill 중앙 관리, 로그 수집, 오류 추적, E2E 테스트 자동화를 고려한 플랫폼 구조를 설계했습니다.

KubernetesStatefulSetRuntime ControllerLoggingPlaywright

Agent는 일반 API가 아니었습니다

  • 사용자별 파일, session, 실행 상태, 로그가 섞이면 안 됩니다.
  • Skill은 공용으로 관리하되, 실행 runtime은 사용자별로 격리해야 합니다.
  • 오류가 났을 때 어떤 사용자의 어떤 runtime에서 발생했는지 추적 가능해야 합니다.
  • UI에서 Agent 실행까지 이어지는 흐름은 단순 API 테스트보다 E2E 테스트가 더 적합했습니다.
Decision사용자 runtime의 고정 identity와 workspace 추적을 위해 StatefulSet 기반 구조를 선택했습니다.

Case 03

Internal Kubernetes Platform for AI Services

서버 발급, 비용, 제한된 네트워크 환경, 관리형 K8s 제약을 우회하기 위해 AI 서비스 전용 Kubernetes 클러스터를 직접 구성하고 운영했습니다.

RKEGitLab CI/CDHarborPrometheusGrafanaLokiIstioLonghorn

왜 직접 클러스터를 만들었나?

  • 서버 발급과 세팅에 시간이 오래 걸리고, 서비스마다 반복되는 환경 세팅 비용이 컸습니다.
  • 제한된 네트워크 환경이라 외부 repository와 binary 사용이 어려워 재현 가능한 설치/배포 방식이 필요했습니다.
  • 여러 AI 서비스를 독립적으로 띄우고, 로그와 metric을 한 곳에서 보고, 운영 이슈를 추적할 수 있어야 했습니다.
Ingress 운영 리스크 대응

기존 annotation 중심 routing을 Gateway API 표준에 맞게 재구성하고 Istio 기반 ingress 구조로 전환했습니다.

일부 요청만 성공하던 현상

서비스 파드 위치에 따라 응답 성공률이 달라지는 현상을 추적해 노드 간 통신 설정 문제를 원인으로 좁혀 조치했습니다.

노드 리소스 사용률 증가

서비스 pod 사용량과 무관하게 특정 노드 리소스가 증가하는 현상을 추적해 감사 로그 프로세스의 과다 동작을 확인하고 안정화했습니다.

Case 04

서로 다른 비즈니스 로직을 가진 5개 영업소의 철강 물류 배차를 위해, ML 예측값과 영업소별 Scorer를 score matrix로 결합하고 운영 가능한 MLOps 구조로 개선했습니다.

FastAPIAutoGluonpandasGitLab CI/CDUnit Test

핵심은 모델보다 변경 가능한 구조였습니다

  • 영업소마다 제품, 지역, 길이, 무게, 하차지, 메모에 따라 차량 유형 선택 규칙이 달랐습니다.
  • 인터뷰 내용과 실제 배차 기록이 다른 경우가 있어 차량 유형을 target으로 하는 multi-class prediction으로 구성했습니다.
  • ML 예측값만으로는 부족하므로, 영업소별 요구사항을 Scorer로 분리해 score matrix에 반영했습니다.

Performance Note

Freight Matching API Optimization

B2C 모바일 서비스의 3초 응답 제한을 맞추기 위해 복화 후보 탐색을 brute-force loop가 아니라 self-merge와 boolean indexing 기반의 테이블 연산으로 재정의했습니다.

FastAPIpandasMSADocker/K8s

기존 배차 기능을 API로 재사용

  • 초기에는 기존 배차 모델 소스 코드를 git submodule로 가져와 재사용했습니다.
  • 하지만 절대 경로, branch 분기, 수정 내역 반영 문제가 생겨 기능 단위 API로 분리하는 방향이 더 안정적이었습니다.
  • 물량별 배차 가능 차량 추론, MEMO 납기일 추출, 복화 매칭을 독립 API로 나누어 재사용성을 높였습니다.
