운영 기반

Main Case Study 01

자동차선 적부 계획 최적화 서비스

자동차선 적부 최적화 모델을 기존 사내 시스템에서 호출 가능한 백엔드 API로 감싸고, 장기 실행 추론을 처리하기 위한 비동기 파이프라인과 분산 상태 관리 구조를 설계했습니다.

Role
API/Backend 설계 · 데이터 모델링 · 모델 연동 · K8s 배포
Scope
요청 검증, 표준 데이터 모델, Engine orchestration, callback 처리, 상태 관리
Stack
Python, FastAPI, Redis, Queue, Kubernetes, Istio
Problem

모델보다 먼저 API 경계가 불안정했습니다.

기존 시스템, API 서버, Engine 서버, 외부 최적화 모델이 서로 다른 데이터 형식을 요구했습니다. 모델 입력 형식에 서비스 전체가 종속되면 모델 교체와 선박 확장 시 변경 범위가 커질 수 있었습니다.

  • 장기 추론 모델은 동기 HTTP 호출로 처리하기 어려웠습니다.
  • API 서버 scale-out 시 callback이 원 요청 pod로 돌아온다는 보장이 없었습니다.
  • 후처리 결과를 기존 시스템이 이해하는 응답 형태로 변환해야 했습니다.
Action

요청, 실행, 상태, 결과 회수를 분리했습니다.

API 서버는 요청 검증과 표준 데이터 모델 변환을 맡고, Engine 서버는 모델 실행 orchestration과 callback 결과 처리를 맡도록 나눴습니다. Redis와 queue를 기준으로 pod와 무관한 상태 조회 흐름을 만들었습니다.

  • 기존 시스템 입력과 모델 입력을 adapter처럼 변환했습니다.
  • job id 기준으로 Redis 상태를 생성하고 callback 처리에 활용했습니다.
  • API pod와 Engine pod를 producer/consumer 구조로 분리했습니다.
Result

운영 시스템이 모델 세부 구현을 몰라도 요청할 수 있게 됐습니다.

요청 생성, 상태 조회, 추론 실행, callback 처리, 후처리 책임이 분리되면서 장기 실행 모델을 운영 API처럼 다룰 수 있게 됐습니다.

  • pod-local state 의존을 제거해 scale-out 이후 callback routing 문제를 완화했습니다.
  • 처리 중, 완료, 실패 상태를 job id와 Redis 기준으로 추적했습니다.
  • 모델 교체와 선박 확장 시 변경 범위를 adapter 계층으로 제한했습니다.
설계 판단

단순 polling이나 pod memory 기반 상태 관리는 scale-out, 장애 복구, 장기 실행 작업에 취약했습니다. 그래서 요청 수신과 결과 회수를 분리하고, Redis를 전역 상태 기준점으로 두는 구조를 선택했습니다.

Main Case Study 02

Personal AI Agent Runtime Platform

개인 AI Agent 서비스의 핵심 백엔드 문제는 LLM 호출 자체보다 사용자별 실행 환경을 어떻게 격리하고 운영할 것인가였습니다. profile, workspace, skill, log, runtime lifecycle이 사용자 사이에서 섞이지 않도록 구조를 분리했습니다.

Role
Platform backend 설계 · runtime lifecycle · service routing
Scope
사용자별 StatefulSet/PVC/Service, provisioning 상태, 공용 skill 동기화
Stack
Kubernetes, StatefulSet, PVC, Service, GitLab CI/CD
Problem

단일 runtime은 사용자 상태와 장애 범위를 섞었습니다.

하나의 runtime에 여러 사용자의 workspace, profile, 파일, 로그를 함께 두면 격리 기준이 흐려졌습니다. runtime 장애도 특정 사용자 문제가 아니라 전체 Agent 서비스 장애처럼 번질 수 있었습니다.

  • 사용자별 실행 디렉터리와 profile을 분리할 기준점이 필요했습니다.
  • runtime 생성, 재시작, 삭제를 서비스 lifecycle로 관리해야 했습니다.
  • 공용 skill 배포와 사용자 runtime 실행 흐름을 분리해야 했습니다.
Action / Result

사용자별 runtime을 Kubernetes 리소스 단위로 고정했습니다.

platform backend가 user id 기준으로 StatefulSet, PVC, Service를 만들고, 채팅 요청은 해당 사용자의 runtime service로 routing하도록 구성했습니다.

  • workspace와 profile 저장소를 runtime 단위로 격리했습니다.
  • runtime 장애나 재시작의 영향을 특정 사용자 범위로 제한했습니다.
  • 공용 skill PVC와 sync job으로 skill 업데이트 영향을 줄였습니다.
설계 판단

Agent를 단일 프로세스형 도구로 보지 않고, 사용자별로 독립 실행되는 backend runtime으로 보았습니다. 그래서 기능 구현보다 먼저 runtime identity, storage, routing, lifecycle의 소유권을 backend에 두었습니다.

Supporting Case Study

Steel Logistics Optimization

철강 물류에서는 복화 후보 탐색을 API 응답 시간 안에 끝내는 계산 구조 문제와, 영업소별로 달라지는 자동배차 기준을 유지보수 가능한 도메인 로직으로 분리하는 문제가 있었습니다.

Role
데이터 처리 구조 개선 · 배차 scoring pipeline 설계
Scope
복화 후보 생성, rule filtering, 차량 유형 예측, 영업소별 scorer
Stack
Python, Pandas, NumPy, ML model, Backend API

Steel Case 01

복화 운송 매칭 성능 개선

출발지와 도착지, 상하차 가능 시간, 차량 유형, 제외 조건을 함께 만족하는 왕복 후보를 찾아야 했습니다. brute-force loop로 모든 조합을 검사하면 물량이 늘어날수록 API 응답 시간 요구사항을 맞추기 어려웠습니다.

Loop 탐색 → Self-merge → Boolean filtering

원본 물량 테이블을 self-merge해 가능한 왕복 후보군을 먼저 만들고, business rule은 boolean mask로 한 번에 필터링했습니다. 복화 후보 계산 시간은 약 900초에서 약 1초 수준으로 줄었습니다.

Steel Case 02

철강 자동배차 구조 개선

자동배차는 제품, 차량 유형, 영업소별 선호도, 현장 제약이 함께 들어가는 도메인 로직 문제였습니다. if/else 중심으로 구현하면 변경이 누적될수록 판단 근거를 설명하기 어려워졌습니다.

차량 유형 예측 → 영업소별 scorer → Score matrix

영업소별 규칙을 scorer 단위로 분리하고, 각 scorer가 산출한 점수를 score matrix로 합성해 배차 판단을 만들도록 구조를 바꿨습니다. 일부 영업소에서는 자동배차 사용률을 80~97% 수준까지 높였습니다.

Supporting Case Study

AI Service Kubernetes Foundation

AI 모델을 운영 서비스로 만들려면 API 코드만으로는 부족했습니다. 폐쇄망 환경에서 이미지 배포, 내부 registry, traffic routing, storage, monitoring을 직접 구성해 여러 AI 서비스를 같은 기준으로 운영할 수 있는 기반을 만들었습니다.

Role
Kubernetes 운영 기반 구성 · 배포/routing/관측 표준화
Scope
CI/CD, 내부 registry, Gateway/Istio routing, storage, monitoring
Stack
Kubernetes, GitLab CI/CD, Helm, Istio, Longhorn, Prometheus, Grafana, Loki
Problem

AI 서비스를 매번 수동 서버 구성으로 늘릴 수 없었습니다.

서버 발급과 클라우드 사용 절차가 느리고, 외부 repository 접근이 제한된 폐쇄망 환경이었습니다. AI API와 모델 서버를 계속 늘리려면 반복 가능한 내부 운영 기반이 필요했습니다.

  • 이미지 빌드, registry push, 배포 흐름을 폐쇄망 안에서 해결해야 했습니다.
  • 서비스별 ingress, routing, network 정책을 일관되게 관리해야 했습니다.
  • 모델 서버 장애와 API 장애를 로그와 metric 기준으로 관측해야 했습니다.
Action / Result

배포, routing, 관측 기준을 Kubernetes 위에 표준화했습니다.

RKE 기반 Kubernetes 클러스터 위에서 AI 서비스 배포에 필요한 registry, CI/CD, storage, monitoring 계층을 정리했습니다. 서비스 traffic은 Gateway/Istio 기준으로 관리하고, 운영 지표와 로그는 Prometheus, Grafana, Loki로 수집했습니다.

  • 신규 AI API와 모델 서버를 반복 가능한 배포 단위로 만들었습니다.
  • 여러 AI 서비스를 같은 클러스터 안에서 독립적으로 운영할 수 있게 했습니다.
  • pod 상태, API 로그, 모델 서버 로그, 장애 징후를 추적할 수 있게 했습니다.
