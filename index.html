Topic 01

선박 공간과 모델 입력 기준 정리

Problem

현업 도면의 적재 구역, 차량 통행 가능성, 선박별 구조 차이를 그대로 API와 모델 입력으로 쓰기 어려웠습니다.

Action

적재 가능 공간은 node, 이동 가능성은 edge로 보고 선박 공간을 graph 구조로 정리했습니다.

Result

선박별 차이를 모델 입력으로 다룰 수 있게 되었고, 이후 선박 확장과 제약 조건 추가가 쉬워졌습니다.

Topic 02

장기 추론과 scale-out 상태 관리

Problem

장시간 실행되는 모델을 동기 HTTP로 호출하기 어려웠고, scale-out 이후 callback이 원 요청 pod로 돌아온다는 보장도 없었습니다.

Action

요청 수신과 결과 수신을 분리하고, queue와 Redis를 기준으로 job 상태를 모든 pod가 공유하도록 바꿨습니다.

Result

pod-local state 의존을 줄이고 처리 중, 완료, 실패 상태를 job id 기준으로 추적할 수 있게 됐습니다.

Topic 03

모델 호출 분산과 추론 시간 단축

Problem

외부 모델 호출이 직렬 loop에 가깝게 동작했고, Kubernetes service의 연결 유지 특성 때문에 특정 pod로 요청이 몰렸습니다.

Action

병렬 가능한 parameter 단위로 모델 호출을 나누고, Istio round-robin routing으로 모델 pod에 요청을 분산했습니다.

Result

적부 추론 시간을 약 10분에서 2~3분 수준으로 줄여 운영 시스템에서 기다릴 수 있는 응답 흐름을 만들었습니다.

Topic 01

사용자별 runtime 격리

Problem

Hermes agent runtime은 디스크의 profile과 사용자 workspace를 폴더 단위로 나눠 격리하고 있었고, 단일 디스크 안에 여러 사용자의 실행 공간이 함께 놓이면 격리 기준과 장애 범위가 흐려질 수 있었습니다.

Action

회원가입 시 user id 기준으로 고유 Hermes pod를 할당하고, StatefulSet, PVC, Service를 생성해 채팅 요청이 해당 사용자의 runtime service 안에서만 처리되도록 routing했습니다.

Result

단일 디스크에서 여러 profile 폴더를 구분하던 방식보다 안전하게 사용자 공간을 격리했고, 장애 범위를 줄이면서 사용자별 runtime을 손쉽게 확장할 수 있는 구조로 바꿨습니다.

Topic 02

채팅 기반 webapp 배포 파이프라인

Problem

현업 사용자가 AI 코딩 도구로 만든 서비스를 다른 사람에게 공유하고 싶어 했지만, 임시 개발 서버에 올린 서비스는 재기동 시 포트와 주소가 바뀌고 로그 확인과 재배포를 우리 팀에 다시 요청해야 했습니다.

Action

채팅으로 webapp을 만들고 배포하는 표준 skill을 만들고, 사용자 id별 GitLab subgroup과 repo를 자동 생성해 Dockerfile과 CI/CD로 Kubernetes에 배포되도록 구성했습니다. 서비스별 내부 proxy를 두어 인가된 사용자만 접근하도록 했습니다.

Result

사용자는 채팅으로 만든 webapp을 안정적인 주소로 배포하고 공유할 수 있었고, 재배포와 로그 확인, 리소스 연결을 Kubernetes와 GitLab CI/CD 기반 흐름으로 표준화했습니다.

Topic 03

공용 skill 관리와 runtime 동기화

Problem

사용자 runtime마다 skill을 따로 관리하면 중복이 커지고, skill 업데이트가 실행 환경 안정성에 직접 영향을 줄 수 있었습니다.

Action

공용 skill 저장소와 runtime sync 흐름을 분리해, 중앙에서 관리한 skill을 사용자 runtime에 동기화하도록 구성했습니다.

Result

skill 관리와 agent 실행 흐름을 분리해 업데이트 영향 범위를 줄이고, 사용자별 runtime 구조를 단순하게 유지했습니다.

Topic 04

Agent harness 표준 작업 흐름 실험

Problem

팀 내 FE/BE 개발 역량 편차가 있었고, 모호한 요구사항을 개발, 검토, 보안 점검, 문서화까지 이어가는 표준 흐름이 필요했습니다.

Action

Main PM이 planner, developer, reviewer, QA, security, document manager에 위임하고 board.md와 task.md로 작업 상태를 공유하게 했습니다.

Result

Supporting Case Study

Steel Logistics Optimization

철강 물류에서는 복화 후보 탐색을 API 응답 시간 안에 끝내야 했고, 자동배차에서는 ML 예측값과 영업소별 판단 기준을 실제 배차 서비스에 적용하는 MLOps 흐름이 필요했습니다.

Role
복화 매칭 최적화 · 자동배차 MLOps/scoring pipeline 설계
Scope
복화 후보 생성, rule filtering, 차량 유형 예측, scorer 운영 흐름
Stack
Python, Pandas, NumPy, ML model, Backend API
Topic 01

복화 운송 매칭 성능 개선

Problem

출발지, 도착지, 시간 조건, 차량 유형, 제외 조건을 모두 만족하는 왕복 후보를 brute-force로 찾으면 약 900초가 걸렸습니다.

Action

조합 탐색을 dataframe join/filtering 문제로 바꾸고, self-merge로 후보군을 만든 뒤 boolean mask로 business rule을 적용했습니다.

Result

복화 후보 계산 시간을 약 900초에서 약 1초 수준으로 줄여 API 응답 시간 요구사항에 맞췄습니다.

Topic 02

철강 자동배차 도메인 로직 구조화

Problem

제품, 차량 유형, 영업소별 선호도, 현장 제약이 if/else로 누적되면 변경 대응과 판단 근거 설명이 어려워졌습니다.

Action

제품별 차량 유형 예측값과 영업소별 scorer를 분리하고, 각 scorer 결과를 score matrix로 합성하도록 구조를 바꿨습니다. 운영 중 모델이 보수적으로 판단하는 구간은 확률값 후처리를 추가해 현업이 기대하는 배차 후보가 더 잘 노출되도록 조정했습니다.

Result

영업소별 요구사항과 모델 출력 보정 기준을 더 빠르게 반영할 수 있었고, 일부 영업소에서는 자동배차 사용률을 80~97% 수준까지 높였습니다.

Topic 02

traffic routing과 ingress 안정화

Problem

서비스별 ingress와 routing 기준이 흩어져 있었고, Ingress NGINX 취약점과 지원 종료 리스크도 함께 관리해야 했습니다.

Action

Result

Topic 03

Problem

모델 서버와 API 서버 장애를 확인하려면 pod 상태뿐 아니라 로그, metric, storage 상태를 함께 볼 수 있어야 했습니다.

Action

Result

Topic 04

운영 장애 분석과 리소스 안정화

Problem

폐쇄망 클러스터 운영 중 네트워크 설정과 OS 보안 로그가 서비스 안정성에 영향을 주는 문제가 발생했습니다.

Action

Result

AI 서비스를 코드 배포뿐 아니라 네트워크, 로그, storage까지 포함한 운영 단위로 다룰 수 있게 됐습니다.
