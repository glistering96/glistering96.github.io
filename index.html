Main Case Study 01

자동차선 적부 계획 최적화 서비스

자동차선 적부 최적화 모델을 기존 사내 시스템에서 호출 가능한 백엔드 API로 감싸고, 장기 실행 추론을 처리하기 위한 비동기 작업 파이프라인과 분산 상태 관리 구조를 설계했습니다. 핵심은 모델을 직접 실행하는 것이 아니라, 운영 시스템이 안정적으로 요청하고 결과를 회수할 수 있는 backend boundary를 만드는 일이었습니다.

문제

백엔드 관점에서 가장 먼저 풀어야 했던 문제는 도메인 복잡도를 API contract로 고정하는 일이었습니다. 기존 시스템, API 서버, Engine 서버, 외부 최적화 모델이 서로 다른 데이터 형식을 요구했기 때문에, 특정 모델 입력 형식에 서비스 전체가 종속되면 모델 교체와 선박 확장이 어려워질 수 있었습니다.

  • 선적지, 양하지, 차량 높이, deck/hold 구조, 통행 가능 경로를 요청 데이터로 안정적으로 표현해야 했습니다.
  • 장기 추론 모델은 동기 HTTP 호출로 처리하기 어려워 timeout, polling, connection 점유 문제가 발생했습니다.
  • API 서버를 scale-out하면 callback 요청이 원 요청을 받은 pod가 아닌 다른 pod로 들어올 수 있었습니다.
  • 요청 상태가 pod memory에 있으면 callback 처리, 실패 추적, 재처리 흐름이 불안정해졌습니다.
  • 모델 결과를 기존 사내 시스템이 이해하는 응답 형태로 변환하는 후처리 계층이 필요했습니다.

해결

API 서버, Engine 서버, 모델 서버의 책임을 분리했습니다. API 서버는 요청 검증, 표준 데이터 모델 변환, 요청 상태 생성을 담당하고, Engine 서버는 모델 실행 orchestration과 callback 결과 처리를 담당하도록 나눴습니다. 모델 서버는 추론 책임만 갖도록 두어 백엔드 계층에서 운영 제어가 가능하게 했습니다.

  • 서비스 내부 표준 데이터 모델을 두고 기존 시스템 입력과 모델 입력을 각각 adapter처럼 변환했습니다.
  • 요청 수신 시 job id와 상태를 생성하고, 추론 요청은 queue에 넣어 API thread가 장시간 점유되지 않게 했습니다.
  • 모델 완료 결과는 callback으로 수신하고, job id 기준으로 Redis에서 상태를 조회해 후처리하도록 구성했습니다.
  • 공용 request queue와 engine queue를 두어 API pod와 Engine pod를 producer/consumer 구조로 분리했습니다.
  • Redis를 전역 상태 저장소로 사용해 어떤 pod가 callback을 받아도 동일한 요청 상태를 조회할 수 있게 했습니다.

결과

결과적으로 모델 실행 자체보다 운영 시스템과 모델 사이의 백엔드 계약이 명확해졌습니다. 요청 생성, 상태 조회, 추론 실행, callback 처리, 결과 후처리가 분리되면서 장기 실행 모델을 운영 API처럼 다룰 수 있게 됐습니다.

  • 기존 사내 시스템은 모델 세부 구현을 몰라도 표준 API로 적부 최적화 요청을 보낼 수 있게 됐습니다.
  • pod-local state 의존을 제거해 scale-out 이후 callback routing 문제를 완화했습니다.
  • job id와 Redis 상태를 기준으로 처리 중, 완료, 실패 상태를 추적할 수 있게 했습니다.
  • 모델 호출 분산과 routing 개선으로 추론 시간을 약 10분에서 2~3분 수준으로 줄였습니다.
  • 표준 데이터 모델을 기준으로 모델 교체와 선박 확장 시 변경 범위를 백엔드 adapter 계층으로 제한했습니다.

Main Case Study 02

Personal AI Agent Runtime Platform

개인 AI Agent 서비스의 핵심 백엔드 문제는 LLM 호출 자체보다 사용자별 실행 환경을 어떻게 격리하고 운영할 것인가였습니다. profile, workspace, skill, log, runtime lifecycle이 사용자 사이에서 섞이지 않도록 Kubernetes 리소스와 platform backend의 책임을 분리했습니다.

문제

단일 runtime에 여러 사용자의 실행 상태를 함께 두면 workspace, profile, 파일, 로그가 섞일 수 있었습니다. 또한 runtime 장애가 특정 사용자 문제로 격리되지 않고 전체 Agent 서비스 장애처럼 번질 수 있었습니다.

  • 사용자별 실행 디렉터리와 profile을 분리할 backend 기준점이 필요했습니다.
  • Agent runtime 생성, 재시작, 삭제를 서비스 lifecycle로 관리해야 했습니다.
  • 공용 skill 배포와 사용자별 runtime 실행 흐름이 서로 영향을 주지 않아야 했습니다.
  • 채팅 요청을 user id 기준으로 올바른 runtime endpoint에 routing해야 했습니다.

해결

platform backend가 사용자별 runtime의 control plane 역할을 맡도록 설계했습니다. 사용자가 Agent를 실행하면 backend가 user id 기준으로 StatefulSet, PVC, Service를 생성하고, 이후 채팅 요청은 해당 사용자의 runtime service로 전달하도록 구성했습니다.

  • 사용자별 StatefulSet/PVC를 두어 workspace와 profile 저장소를 runtime 단위로 격리했습니다.
  • runtime provisioning, endpoint 조회, lifecycle 제어를 platform backend 책임으로 고정했습니다.
  • 공용 skill PVC와 sync job으로 skill 배포 흐름을 사용자 runtime 실행 흐름과 분리했습니다.
  • 사용자별 service routing을 통해 동일한 Web/Chat UI에서 각자의 Agent runtime으로 요청을 전달했습니다.

결과

Agent 서비스를 단일 프로세스형 도구가 아니라 사용자별로 독립 실행되는 backend runtime 플랫폼으로 정리했습니다. 실행 상태와 파일 시스템이 사용자 단위로 분리되면서 장애 범위, 로그 추적, runtime lifecycle 관리가 명확해졌습니다.

  • 사용자별 Agent 실행 환경을 Kubernetes 리소스 단위로 격리했습니다.
  • runtime 장애나 재시작의 영향을 특정 사용자 범위로 제한할 수 있는 구조를 확보했습니다.
  • skill 업데이트와 runtime 실행을 분리해 운영 중 변경 영향을 줄였습니다.
  • platform backend가 runtime 생성부터 routing까지 제어하는 구조를 만들었습니다.

Supporting Case Study

Steel Logistics Optimization

철강 물류에서는 두 종류의 백엔드 문제가 있었습니다. 하나는 복화 후보 탐색을 API 응답 시간 안에 끝내는 계산 구조 문제였고, 다른 하나는 영업소별로 달라지는 자동배차 기준을 유지보수 가능한 도메인 로직으로 분리하는 문제였습니다.

Steel Case 01

복화 운송 매칭 성능 개선

조합 탐색 문제를 dataframe join/filtering 문제로 재정의해 계산 시간을 약 900초에서 약 1초 수준으로 줄였습니다.

문제

복화 매칭은 출발지와 도착지, 상하차 가능 시간, 차량 유형, 불가능한 조합 조건을 함께 만족하는 왕복 후보를 찾아야 했습니다. brute-force loop로 모든 조합을 검사하면 물량이 늘어날수록 API 응답 시간 요구사항을 맞추기 어려웠습니다.

  • 50개 이상 물량에서 후보 탐색 시간이 약 15분까지 늘어났습니다.
  • B2C 성격의 화면에서는 수 초 안에 후보를 반환해야 했습니다.
  • multiprocessing, batch 분할, C++ 전환은 운영 복잡도와 변환 비용이 컸습니다.

해결

후보 탐색을 Python loop가 아니라 dataframe 연산 문제로 바꿨습니다. 원본 물량 테이블을 self-merge해 가능한 왕복 후보군을 먼저 만들고, business rule은 boolean indexing 조건으로 한 번에 필터링했습니다.

  • self-merge로 상행/하행 후보 조합을 vectorized operation으로 생성했습니다.
  • 시간 조건, 출발지/도착지 조건, 차량 유형 조건, 제외 조건을 boolean mask로 구성했습니다.
  • 후보군 생성과 필터링을 pandas 연산으로 밀어 넣어 API 병목을 줄였습니다.

결과

복화 매칭 병목을 알고리즘 구현 방식이 아니라 데이터 처리 구조 관점에서 해결했습니다. 서비스 응답 시간 안에서 후보를 반환할 수 있게 되었고, 조건 추가도 boolean rule 단위로 확장할 수 있게 됐습니다.

  • 복화 후보 계산 시간을 약 900초에서 약 1초 수준으로 단축했습니다.
  • API 응답 시간 요구사항을 만족할 수 있는 계산 구조를 확보했습니다.
  • 추가 business rule을 dataframe filter 조건으로 반영할 수 있게 했습니다.

Steel Case 02

철강 자동배차 구조 개선

영업소별로 다른 배차 기준을 scorer와 score matrix 구조로 분리해 변경 대응성과 판단 근거 추적성을 높였습니다.

문제

자동배차는 제품, 차량 유형, 영업소별 선호도, 현장 제약이 함께 들어가는 도메인 로직 문제였습니다. 영업소마다 요구사항이 달라 if/else 중심으로 구현하면 변경이 누적될수록 배차 판단 근거를 설명하기 어려워졌습니다.

  • 제품별로 배차 가능한 차량 유형이 달라 사전 분류 기준이 필요했습니다.
  • 영업소별 배차 기준이 계속 바뀌어 로직을 빠르게 교체할 수 있어야 했습니다.
  • 배차 결과가 왜 선택됐는지 score 단위로 추적할 수 있는 구조가 필요했습니다.

해결

제품별 배차 가능 차량 유형을 먼저 예측하고, 영업소별 business logic은 scorer 객체로 분리했습니다. 각 scorer가 산출한 점수를 score matrix로 합성해 배차 판단을 만들도록 구조를 바꿨습니다.

  • 제품 속성 기반 차량 유형 분류를 multi-class prediction 문제로 정리했습니다.
  • 영업소별 규칙을 scorer 단위로 분리해 조건 추가와 교체 범위를 줄였습니다.
  • score matrix로 후보별 판단 근거를 남겨 디버깅과 현업 설명이 가능하게 했습니다.

결과

자동배차 로직을 단일 조건문 덩어리가 아니라 교체 가능한 scoring pipeline으로 정리했습니다. 현업 요구사항 변경 시 전체 배차 흐름을 흔들지 않고 scorer 단위로 대응할 수 있는 구조를 확보했습니다.

  • 영업소별 business rule을 scorer 단위로 모듈화했습니다.
  • 제품별 차량 유형 예측값과 도메인 score를 함께 사용하는 배차 구조를 만들었습니다.
  • 일부 영업소에서 자동배차 사용률을 80~97% 수준까지 높였습니다.

Supporting Case Study

AI Service Kubernetes Foundation

AI 모델을 운영 서비스로 만들려면 API 코드만으로는 부족했습니다. 폐쇄망 환경에서 이미지 배포, 내부 registry, traffic routing, storage, monitoring을 직접 구성해 여러 AI 서비스를 같은 기준으로 운영할 수 있는 기반을 만들었습니다.

문제

서버 발급과 클라우드 사용 절차가 느리고, 외부 repository 접근이 제한된 폐쇄망 환경이었습니다. AI API와 모델 서버를 계속 늘리려면 매번 수동 서버 구성에 의존하지 않는 내부 운영 기반이 필요했습니다.

  • 이미지 빌드, registry push, 배포 흐름을 폐쇄망 안에서 해결해야 했습니다.
  • 서비스별 ingress, routing, network 정책을 일관되게 관리해야 했습니다.
  • 모델 서버 장애와 API 장애를 로그와 metric 기준으로 관측할 수 있어야 했습니다.

해결

RKE 기반 Kubernetes 클러스터를 구성하고, AI 서비스 배포에 필요한 registry, CI/CD, storage, monitoring 계층을 함께 정리했습니다. 서비스 traffic은 Gateway/Istio 기준으로 관리하고, 운영 지표와 로그는 Prometheus, Grafana, Loki로 수집하도록 구성했습니다.

  • GitLab CI/CD, Harbor, Helm으로 API 서버와 모델 서버 이미지를 독립 배포할 수 있게 했습니다.
  • Istio Gateway 기반 routing으로 서비스별 traffic 경계를 정리했습니다.
  • Longhorn, Prometheus, Grafana, Loki를 이용해 저장소와 관측 기반을 구성했습니다.

결과

AI 서비스를 일회성 서버 배포가 아니라 반복 가능한 Kubernetes 운영 단위로 다룰 수 있게 됐습니다. 신규 AI API나 모델 서버를 추가할 때 배포, routing, 관측 기준을 재사용할 수 있는 내부 기반을 확보했습니다.

  • 폐쇄망 환경에서 AI 서비스 이미지를 빌드, 저장, 배포하는 흐름을 만들었습니다.
  • 여러 AI 서비스를 같은 클러스터 안에서 독립적으로 운영할 수 있게 했습니다.
  • pod 상태, API 로그, 모델 서버 로그, 장애 징후를 추적할 수 있는 관측 체계를 구성했습니다.
