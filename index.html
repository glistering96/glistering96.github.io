Focus

제가 반복해서 해결해 온 문제예요.

포트폴리오를 프로젝트 순서가 아니라 백엔드 시스템 문제 기준으로 다시 묶었습니다. 핵심은 API, queue, data contract, runtime, observability, deployment입니다.

01

장시간 작업 처리

동기 HTTP로 처리하기 어려운 AI 추론을 callback과 queue 기반 작업 흐름으로 분리했습니다.

02

분산 상태 관리

scale-out 이후 pod local state에 의존하지 않도록 요청 상태와 결과를 외부 저장소로 분리했습니다.

03

사용자별 런타임 격리

AI Agent가 사용하는 workspace, session, logs를 사용자 단위로 격리하는 구조를 설계했습니다.

04

운영 가능한 플랫폼

폐쇄망에서 AI 서비스를 배포하고 관찰하기 위한 Kubernetes, CI/CD, monitoring 기반을 구축했습니다.

Case Studies

대표 백엔드 사례

Case 01

자동차선 적부 최적화 Backend Service

5~20분 AI 추론을 기존 사내 시스템과 연결한 비동기 백엔드 구조

FastAPIQueuePydanticKubernetes

문제

추론 시간이 길고 편차가 커서 동기 응답으로는 timeout과 resource blocking 위험이 컸습니다.

설계

API 서버와 engine 서버를 분리하고 request queue, result store, callback processor로 흐름을 나눴습니다.

결과

모델을 연구 코드가 아니라 기존 시스템과 연동 가능한 서비스 경계로 전환했습니다.

Internal SystemAPIQueueEngineResult StoreCallback

Case 02

Personal AI Agent Runtime Platform

사용자별 격리 runtime을 제공하는 사내 Agent 실행 플랫폼

K8sStatefulSetLoggingPlaywright

문제

Agent는 stateless API가 아니라 사용자별 workspace, session, file, logs를 가진 실행 환경이 필요했습니다.

설계

backend control layer가 사용자 runtime을 관리하고, StatefulSet 기반으로 고정 identity를 유지하게 했습니다.

결과

공용 skill은 중앙 관리하고 실행 환경은 사용자별로 격리하는 플랫폼 구조를 만들었습니다.

Case 03

Internal Kubernetes Platform for AI Services

폐쇄망 환경에서 AI 서비스를 배포하고 운영하기 위한 내부 플랫폼

RKEGitLab CI/CDHarborGrafanaLoki

문제

서버 발급, 세팅, 폐쇄망 제약 때문에 AI 서비스를 빠르게 독립 배포하기 어려웠습니다.

설계

RKE 기반 K8s, Harbor, GitLab Runner, Helm, monitoring stack으로 서비스 운영 기반을 구성했습니다.

운영

Ingress 보안 리스크, 노드 간 방화벽, SELinux audit logging 문제를 원인까지 추적해 조치했습니다.

Case 04

Steel TMS MLOps Backend

ML 추론과 영업소별 도메인 규칙을 결합한 자동배차 백엔드

FastAPIAutoGluonpandasGitLab

문제

영업소별 배차 규칙이 다르고, 현업 인터뷰와 실제 데이터가 일치하지 않는 경우가 많았습니다.

설계

차량 유형 예측값과 Scorer 기반 business logic을 score matrix로 통합했습니다.

결과

20% → 80% 사용률 개선, 일부 영업소 97%까지 상승했습니다.

Performance Note

900초 걸리던 복화 매칭을 1초 수준으로 줄였어요.

B2C 모바일 서비스의 3초 응답 제한을 맞추기 위해 문제를 후보쌍 생성과 조건 필터링으로 재정의했습니다. self-merge로 가능한 후보쌍을 만들고 boolean indexing으로 불가능한 매칭을 제거했습니다.

+ boolean indexing
