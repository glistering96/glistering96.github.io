Main Case Study 01

자동차선 적부 계획 최적화 서비스

자동차선 적부 최적화 모델을 기존 사내 시스템에서 호출 가능한 백엔드 API로 감싸고, 장기 실행 추론을 처리하기 위한 비동기 파이프라인과 분산 상태 관리 구조를 설계했습니다.

API/Backend 설계 · 데이터 모델링 · 모델 연동 · K8s 배포
Scope
요청 검증, 표준 데이터 모델, Engine orchestration, callback 처리, 상태 관리

Topic 01

선박 공간과 모델 입력 기준 정리

Problem

현업 도면의 적재 구역, 차량 통행 가능성, 선박별 구조 차이를 그대로 API와 모델 입력으로 쓰기 어려웠습니다.

Action

적재 가능 공간은 node, 이동 가능성은 edge로 보고 선박 공간을 graph 구조로 정리했습니다.

Result

선박별 차이를 모델 입력으로 다룰 수 있게 되었고, 이후 선박 확장과 제약 조건 추가가 쉬워졌습니다.

Topic 02

시스템 간 데이터 경계 분리

Problem

기존 시스템, API 서버, Engine 서버, 외부 모델이 서로 다른 데이터 형식을 요구해 변경 영향이 커질 수 있었습니다.

Action

서비스 내부의 표준 데이터 모델을 두고 각 시스템과 모델은 adapter 계층에서 입출력을 변환하도록 설계했습니다.

Result

운영 시스템이 모델 세부 구현을 몰라도 요청할 수 있고, 모델 교체 시 변경 범위를 변환 계층으로 제한했습니다.

Topic 03

장기 추론과 scale-out 상태 관리

Problem

장시간 실행되는 모델을 동기 HTTP로 호출하기 어려웠고, scale-out 이후 callback이 원 요청 pod로 돌아온다는 보장도 없었습니다.

Action

요청 수신과 결과 수신을 분리하고, queue와 Redis를 기준으로 job 상태를 모든 pod가 공유하도록 바꿨습니다.

Result

pod-local state 의존을 줄이고 처리 중, 완료, 실패 상태를 job id 기준으로 추적할 수 있게 됐습니다.

Topic 04

모델 호출 분산과 추론 시간 단축

Problem

외부 모델 호출이 직렬 loop에 가깝게 동작했고, Kubernetes service의 연결 유지 특성 때문에 특정 pod로 요청이 몰렸습니다.

Action

병렬 가능한 parameter 단위로 모델 호출을 나누고, Istio round-robin routing으로 모델 pod에 요청을 분산했습니다.

Result

적부 추론 시간을 약 10분에서 2~3분 수준으로 줄여 운영 시스템에서 기다릴 수 있는 응답 흐름을 만들었습니다.

Topic 01

사용자별 runtime 격리

Problem

단일 runtime에 여러 사용자의 workspace, profile, 파일, 로그가 섞이면 격리 기준과 장애 범위가 흐려졌습니다.

Action

user id 기준으로 StatefulSet, PVC, Service를 생성하고 채팅 요청은 해당 사용자의 runtime service로 routing했습니다.

Result

workspace와 profile을 사용자 runtime 단위로 분리하고, 장애나 재시작 영향을 특정 사용자 범위로 제한했습니다.

Topic 02

runtime lifecycle과 실행 추적

Problem

runtime 생성, 재시작, 삭제가 임시 작업처럼 흩어지면 사용자의 실행 환경 상태와 오류 원인을 추적하기 어려웠습니다.

Action

platform backend에서 provisioning 상태, runtime identity, service routing, execution log를 사용자 기준으로 관리했습니다.

Result

user, runtime, execution 기준으로 실행 흐름과 오류를 확인할 수 있어 운영 가능한 backend resource처럼 다룰 수 있게 됐습니다.

Topic 03

공용 skill 관리와 runtime 동기화

Problem

사용자 runtime마다 skill을 따로 관리하면 중복이 커지고, skill 업데이트가 실행 환경 안정성에 직접 영향을 줄 수 있었습니다.

Action

공용 skill 저장소와 runtime sync 흐름을 분리해, 중앙에서 관리한 skill을 사용자 runtime에 동기화하도록 구성했습니다.

Result

skill 관리와 agent 실행 흐름을 분리해 업데이트 영향 범위를 줄이고, 사용자별 runtime 구조를 단순하게 유지했습니다.

Topic 04

Agent harness 표준 작업 흐름 실험

Problem

팀 내 FE/BE 개발 역량 편차가 있었고, 모호한 요구사항을 개발, 검토, 보안 점검, 문서화까지 이어가는 표준 흐름이 필요했습니다.

Action

Main PM이 planner, developer, reviewer, QA, security, document manager에 위임하고 board.md와 task.md로 작업 상태를 공유하게 했습니다.

Result

K8s 이관 코드와 인프라 리소스 작성, secret hardcoding 점검, auto-stowage 강화학습 agent 실험까지 빠르게 확장했습니다.

Supporting Case Study

Steel Logistics Optimization

철강 물류에서는 복화 후보 탐색을 API 응답 시간 안에 끝내는 계산 구조 문제와, 영업소별로 달라지는 자동배차 기준을 유지보수 가능한 도메인 로직으로 분리하는 문제가 있었습니다.

Role
데이터 처리 구조 개선 · 배차 scoring pipeline 설계
Scope
복화 후보 생성, rule filtering, 차량 유형 예측, 영업소별 scorer
Stack
Python, Pandas, NumPy, ML model, Backend API
Topic 01

복화 운송 매칭 성능 개선

Problem

출발지, 도착지, 시간 조건, 차량 유형, 제외 조건을 모두 만족하는 왕복 후보를 brute-force로 찾으면 약 900초가 걸렸습니다.

Action

조합 탐색을 dataframe join/filtering 문제로 바꾸고, self-merge로 후보군을 만든 뒤 boolean mask로 business rule을 적용했습니다.

Result

복화 후보 계산 시간을 약 900초에서 약 1초 수준으로 줄여 API 응답 시간 요구사항에 맞췄습니다.

Topic 02

철강 자동배차 도메인 로직 구조화

Problem

제품, 차량 유형, 영업소별 선호도, 현장 제약이 if/else로 누적되면 변경 대응과 판단 근거 설명이 어려워졌습니다.

Action

제품별 차량 유형 예측값과 영업소별 scorer를 분리하고, 각 scorer 결과를 score matrix로 합성하도록 구조를 바꿨습니다.

Result

영업소별 요구사항을 더 빠르게 반영할 수 있었고, 일부 영업소에서는 자동배차 사용률을 80~97% 수준까지 높였습니다.

Topic 01

폐쇄망 AI 서비스 배포 기반 구성

Problem

서버 발급과 클라우드 사용 절차가 느리고 외부 repository 접근이 제한되어, AI API와 모델 서버를 반복 배포하기 어려웠습니다.

Action

RKE 기반 Kubernetes 클러스터에 내부 registry, GitLab CI/CD, Helm 배포 흐름을 구성했습니다.

Result

신규 AI API와 모델 서버를 같은 기준의 이미지, manifest, 배포 단위로 반복 운영할 수 있게 됐습니다.

Topic 02

traffic routing과 ingress 안정화

Problem

서비스별 ingress와 routing 기준이 흩어져 있었고, Ingress NGINX 취약점과 지원 종료 리스크도 함께 관리해야 했습니다.

Action

Istio와 Gateway API 기반으로 traffic 진입점을 정리하고, 노드 간 방화벽 설정 누락으로 발생한 ingress 장애를 분석해 조치했습니다.

Result

여러 AI 서비스를 같은 클러스터 안에서 독립적으로 노출하고 routing 정책을 일관되게 관리할 수 있게 됐습니다.

Topic 03

storage와 monitoring 운영 기준 정리

Problem

모델 서버와 API 서버 장애를 확인하려면 pod 상태뿐 아니라 로그, metric, storage 상태를 함께 볼 수 있어야 했습니다.

Action

Longhorn 기반 storage와 Prometheus, Grafana, Loki 기반 관측 구성을 붙이고 서비스별 로그와 metric을 수집했습니다.

Result

pod 상태, API 로그, 모델 서버 로그, 장애 징후를 같은 운영 화면에서 추적할 수 있는 기반을 확보했습니다.

Topic 04

운영 장애 분석과 리소스 안정화

Problem

폐쇄망 클러스터 운영 중 네트워크 설정과 OS 보안 로그가 서비스 안정성에 영향을 주는 문제가 발생했습니다.

Action

ingress traffic 장애 원인을 방화벽 설정에서 찾고, SELinux audit logging 과다 점유 문제를 분석해 운영 리소스 사용을 조정했습니다.

Result

AI 서비스를 코드 배포뿐 아니라 네트워크, 로그, storage까지 포함한 운영 단위로 다룰 수 있게 됐습니다.
