Supporting Case

Internal Kubernetes Platform for AI Services

제한된 네트워크 환경에서도 AI 서비스를 독립된 서비스 단위로 배포·관찰·운영할 수 있도록 내부 Kubernetes 플랫폼을 설계한 사례입니다.

기간

2025.01 ~ 현재

역할

플랫폼 구성, CI/CD 운영 체계 수립, 장애 대응 프로세스 정비

기술 스택

Rancher, Helm, GitLab CI/CD, Harbor, Grafana, Prometheus, Loki

핵심 성과

보안 대응·네트워크 제약·운영 재현성 이슈를 플랫폼 단위로 정리

운영 포인트

Ingress 보안 전환, 노드 통신 정합성, SELinux 로그 과다 제어

Context

왜 플랫폼이 먼저였는지

제한된 네트워크 환경에서는 외부 의존도가 높을수록 배포 속도가 느려집니다. 게다가 AI 서비스는 배포 단위가 늘어날수록 운영 규칙을 매번 새로 정의하기 어렵습니다. 그래서 특정 서비스마다 따로 관리하지 않고 공통 플랫폼 템플릿을 먼저 고정했습니다.

Backend Problem

AI 서비스 운영에서의 병목

문제

  • Ingress/보안 정책과 배포 정책이 서비스마다 다르게 운용됨
  • 노드 통신 문제가 누락되어 특정 경로만 실패
  • 로그/메트릭 수집이 분산되어 장애 원인 파악이 느림

해결

  • 클러스터 템플릿을 표준화하고 GitLab CI/CD로 자동화
  • 서비스 런타임/네트워크/관측성 계층을 일원화
  • 장애 대응 표준을 runbook로 고정
Architecture / Figure

플랫폼 구조

플랫폼 개요

이 그림에서 봐야 할 점

  • 서비스 단위로 분리된 배포 단위를 공통 템플릿으로 관리
  • 제한된 네트워크에서도 이미지/배포 자원 경로를 표준화
  • 운영/개발용 관측 지표를 분리 조회할 수 있게 구성

트러블슈팅 포인트

이 그림에서 봐야 할 점

  • 보안 경로가 바뀔 때 ingress/서비스 라우팅이 함께 안정화되어야 함
  • 노드 간 통신 누락은 일부 요청만 통과되는 장애로 이어질 수 있음
  • 관측성 노이즈를 줄여야 실제 문제 알람만 남음
Key Design Decisions (ADR)

ADR

ADR-01 / Rancher 선택

Decision
클러스터 운영의 반복 작업을 Rancher 기반 템플릿으로 고정
Context
서비스별 운영 방식이 달라 운영 부담이 누적됨
Why
배포·권한·헬스체크 규칙의 공통화를 통해 운영 재현성을 확보
Trade-off
도입 초기 학습비용이 발생하지만 운영 단가가 하향

ADR-02 / Gateway API 전환

Decision
Ingress NGINX에서 Istio + Gateway API로 라우팅 체계 전환
Context
보안 패치 대응 속도 및 트래픽 균형 제어가 필요
Why
보안 대응을 구조적으로 반영하고 트래픽 분산 실패를 줄이기 위해
Trade-off
기존 경로 설정을 새 표준으로 재작성해야 함

ADR-03 / 관측성 우선

Decision
로그·메트릭·알림을 서비스 공통 스키마로 통합
Context
장애 분석에 필요한 지표 산출이 일관되지 않음
Why
문제 조기 발견 및 운영 인수인계를 체계화
Trade-off
초기 계측 및 대시보드 정합성 작업이 큼
Troubleshooting / Operation

운영 대응 표

증상 원인 조치 결과
일부 요청만 성공하고 일부는 실패 Ingress NGINX의 위험 구간 노출과 보안 정책 변경 이슈 Istio + Gateway API로 ingress 구조 전환 보안 대응 속도와 라우팅 일관성이 개선
요청이 특정 node로만 집중 노드 통신 정책 및 포트 설정 누락 방화벽·포트 정책을 노드 공통 템플릿으로 통일 요청 분산이 안정화되고 실패 구간이 축소
특정 node의 메모리/CPU 비정상 증가 SELinux 감사 로그 과다 및 데몬 동작 이슈 로그 수집 정책 재조정과 모니터링 임계치 튜닝 node 자원 압박이 완화되고 알림 노이즈가 감소
Results

결과

운영성

서비스별 배포 규칙을 템플릿화해 반복 운영 작업을 줄였습니다.

보안 대응

보안 취약점 반영 과정을 아키텍처 차원으로 흡수했습니다.

장애 대응

장애 유형별 runbook가 정리되어 인수인계가 쉬워졌습니다.

Lessons Learned

교훈

플랫폼 우선 원칙

서비스 단위 최적화보다 플랫폼 표준을 먼저 두는 것이 운영 비용을 낮춥니다.

보안은 아키텍처

보안 조치는 패치가 아니라 트래픽 구조와 라우팅 설계에 반영되어야 합니다.

기록의 힘

장애 이슈를 “증상-원인-조치-결과”로 남겨야 축적 가능한 운영 조직으로 이동합니다.
