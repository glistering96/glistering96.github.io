Supporting Case

Personal AI Agent Runtime Platform

사용자별 Agent 실행 환경을 격리하고, skill과 log를 중앙에서 관리할 수 있는 multi-tenant runtime platform backend를 설계했습니다.

핵심 메시지

AI 모델을 하나의 서비스로 보는 대신 사용자별 runtime를 분리해 운영 책임을 확장했습니다.

기간

2024.04 ~ 2025.10 (운영 고도화 기간 포함)

역할

runtime 설계, 로그 수집 구조 설계, 배포/운영 정책 수립

기술

Kubernetes StatefulSet, 공용 skill catalog, API gateway, 중앙 로그 수집

성과

사용자별 실행 독립성을 확보하고 운영 추적성을 일관된 방식으로 축적

Context

왜 개인 runtime 분리가 필요한가

내부 PoC 단계에서 동일 컨테이너/프로세스에서 여러 사용자의 작업을 처리하면 의존성 충돌, 권한 경계, 작업별 상태 섞임이 발생했습니다. 또한 로그 분석 시 어떤 사용자 작업이 실패했는지 분리하기 어려워 운영자 개입 비용이 커졌습니다.

Backend Problem

일반 API 구성만으로는 부족했던 점

문제

  • 공유 실행 환경에서 사용자별 설정이 섞이면 런타임 재현성이 깨짐
  • skill의 버전/권한 정책을 중심으로한 중앙 제어가 어려움
  • 요청 로그에 user-context와 agent 실행 context가 결합되지 않음

방식

  • 사용자별 StatefuleSet 기반 실행 단위를 설계
  • skill catalog를 API로 중앙 관리하고 실행 환경에 배포
  • 로그 수집기를 모든 실행 단위에 주입해 공용 저장소로 집계
Architecture / Figure

도식으로 보는 핵심 설계

사용자별 runtime 중심 구조

이 그림에서 봐야 할 점

  • 사용자별로 격리된 runtime 단위를 갖추어 충돌과 상태 혼입을 방지
  • 실행 단위를 기준으로 skill/권한/리소스를 다르게 배치
  • 로그는 execution id 단위로 공용 로그 저장소에 적재되어 추적 가능

운영 플랫폼 레이어

이 그림에서 봐야 할 점

  • 서비스/런타임/관측성 계층을 분리해 장애 전파 범위를 제한
  • 환경별 구성 파라미터는 values 기반으로 관리해 운영 일관성 확보
  • 런타임 신규 추가 시 제어면의 변경만으로 온보딩
Key Design Decisions (ADR)

ADR 카드

ADR-01

Decision
멀티테넌트 격리를 위해 사용자별 runtime 단위를 StatefulSet로 운영
Context
공유 환경에서는 사용자 간 설정 충돌이 잦고 상태 추적이 불명확
Why
고가용성과 복구 시 user-level 격리를 보장하기 위해
Trade-off
동일 기능의 중복 pod 운영으로 리소스 사용량이 증가

ADR-02

Decision
Skill 관리와 실행 권한을 API 기반 catalog에서 중앙 제어
Context
개별 런타임마다 스크립트가 달라 버전 일관성이 깨짐
Why
보안과 운영 규칙을 일괄 적용하기 위해
Trade-off
catalog 갱신 주기와 캐시 무효화 정책이 추가로 필요

ADR-03

Decision
실행 로그를 request-context-id 기준으로 집계
Context
로그가 분산되어 장애 루트를 추적하기 어려움
Why
장애 응답 속도와 분석 정확성을 높이기 위해
Trade-off
PII 마스킹 규칙과 저장소 비용을 함께 설계해야 함
Implementation Details

실제 구현 포인트

API Boundary

  • 사용자 생성/삭제/재시작 API와 런타임 제어 API 분리
  • runtime 상태 조회는 일관된 response schema로 제공
  • skill 배포 API는 버전, 대상 사용자 집합을 명시적으로 받음

Runtime / Isolation

  • 사용자당 고정 storage와 설정 경로를 주어 상태 분리
  • 리소스 제한을 프로파일로 구분해 과점유 방지
  • 장애 pod는 자동 재생성 후 최신 catalog 동기화 수행

Central Skill & Logs

  • skill 사용 이력은 platform db에 기록
  • 실행 로그/에러 로그를 공용 수집기로 전송
  • 요청 단위 traceId로 실행 경로를 일관되게 추적

Monitoring

  • 런타임 시작/종료, 메모리/CPU 임계치 알림을 설정
  • 동시실행 상한 초과 시 신규 runtime 생성을 제어
  • 운영 로그를 기반으로 실행 실패 규칙을 분류하고 알림 정책 반영
Troubleshooting / Operation

운영 대응표

증상 원인 조치 결과
사용자 A의 스킬 변경이 사용자 B에 영향을 줌 공유 실행 환경에서 스킬 바인딩이 전역 적용됨 사용자별 런타임 분리 및 catalog binding 단계를 분리 격리성이 회복되고 충돌 이슈가 감소
실패 로그에서 작업 대상 사용자를 빠르게 식별하기 어려움 로그에 execution context가 충분히 포함되지 않음 요청/에이전트 id 기반 공통 로그 스키마 적용 장애 진단 시간이 단축됨
초기 onboarding 시 스킬/권한이 즉시 반영되지 않음 배포 동기화 시점이 분산 runtime restart 정책과 catalog 조회 캐시 정책 정립 반영 지연이 줄고 운영 요청이 감소

skill 배포와 runtime 제어를 API화해 수동 개입 지점을 줄였습니다.

Lessons Learned

교훈

멀티테넌시 운영

격리 전략은 보안만의 문제가 아니라, 장애 확산 범위를 줄이는 운영 안정성 장치였습니다.

플랫폼 계약

Runtime이 바뀌어도 catalog/API 계약이 흔들리지 않도록 하는 설계가 지속적 확장성의 핵심입니다.

모니터링 우선순위

시작/종료/실패의 최소 이벤트를 먼저 정교화하면 추가 기능의 디버깅 비용이 크게 줄어듭니다.
