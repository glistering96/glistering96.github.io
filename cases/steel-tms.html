Supporting Case

Steel TMS MLOps Backend

사용자와 영업 조직의 운용 암묵지를 반영한 Scorer/Score Matrix 구조로 배차 예측과 API 운영을 연결해, 운영 가능한 MLOps 백엔드로 정비한 사례입니다.

프로젝트 요약

현업 규칙 기반 배차 기준을 ML multi-class 예측과 정책 점수 행렬로 추상화해 의사결정 엔진을 정리했습니다.

기간

2024.02 ~ 2024.11 (고도화 운영 반영)

역할

도메인 규칙 정리, Scorer 설계, API 경계 분리, 운영 표준화

기술 스택

Python, FastAPI, pandas, Redis, GitLab CI/CD, Grafana/Prometheus, Helm

정량 성과

요건 반영 속도 개선: 1주 내 요구반영, 배차 사용률 20% → 80%, 일부 영업소 97% 도달

Context

왜 이 문제가 중요했는가

기존 배차 판단은 담당자별 규칙 지식에 의존해 일관성이 낮았고, 정책 변경 시 QA/릴리즈 부담이 컸습니다. 또한 영업소마다 운송 가능 차량군과 우선순위가 달라, 동일한 후보를 다르게 판단해야 하는 상황이 반복되어 운영 정책이 늘어졌습니다.

Backend Problem

단순 API/CRUD로는 부족했던 이유

주요 병목

  • 규칙이 코드와 DB에 흩어져 운영자가 예측 경로를 빠르게 이해하기 어려움
  • 요청마다 정책 계산이 분산되어 응답 스키마와 로그 규격이 일정하지 않음
  • 재학습/재배포 주기가 느려 현업 요구 반영이 지연됨
  • 관측성 규약이 없어 배차 실패 원인을 추적하는데 리소스 과다 소요

해결 포인트

  • 예측값과 규칙 점수를 분리한 API boundary 정의
  • 영업소별 정책을 score matrix로 정규화
  • 예측 결과, 배차 규칙, 최종 스코어를 독립적으로 저장하고 추적 가능하게 구성
  • CI/CD와 모니터링을 붙여 변경 후 운영 영향의 가시화 확보
Architecture / Figure

Figure로 보는 핵심 구조

차량 매핑 및 예측값 생산

이 그림에서 봐야 할 점

  • 학습 예측값을 그대로 노출하지 않고 backend scoring 입력으로 표준화
  • 예측 불확실성까지 로그에 남겨 모델 편차 분석에 반영
  • 현업 정책 변경이 있어도 예측값 계층은 유지 가능

Scorer / Score Matrix

이 그림에서 봐야 할 점

  • 예측 점수와 정책 점수를 분리해 조합 가중치를 명시적으로 관리
  • 영업소별 요구를 행렬 구성 값으로 관리해 정책 교체를 단순화
  • 문제 발생 시 정책 이력까지 재생성 가능해 운영 감사성이 좋아짐
Key Design Decisions (ADR)

ADR 카드

ADR-01

Decision
multi-class 예측 API와 Scorer API를 분리
Context
둘 다 결과를 단일 응답으로 묶을 경우 계약 변경 충격이 커짐
Why
예측 변경과 운영 정책 변경을 독립적으로 배포하기 위함
Trade-off
조합 단계가 늘어나 초기 설계 오버헤드 증가

ADR-02

Decision
영업소별 룰을 score matrix로 집중 관리
Context
코드 분기와 SQL 분기 방식으로 규칙 관리가 누적됨
Why
운영 정책 변경 반영 시간을 줄이고 감사 이력을 남기기 위함
Trade-off
초기에는 행렬 스키마 설계와 관리 체계가 필요

ADR-03

Decision
GitLab CI/CD와 모니터링 파이프라인을 강제 경로로 고정
Context
수동 배포 방식은 정책 변경 시 영향 분석이 늦어졌음
Why
1주 내 요구 반영이라는 운영 SLA를 지키기 위함
Trade-off
빌드/배포 자동화 규칙이 생기면서 파이프라인 유지보수 비용이 상승
Implementation Details

구현 포인트

API Boundary

  • 배차 후보 생성, 후보 필터링, 최종 점수 산정 API를 분리
  • 예측 결과와 규칙 적용 결과를 다른 코드 경로로 추적 가능하도록 응답 스키마 분리
  • 요청별 idempotency 키로 중복 점수 계산을 방지

Data Model & 계약

  • 예측 점수, 규칙 점수, 최종 점수 DTO를 구분해 내부 계약 고정
  • 영업소별 score matrix 버전 정보를 헤더에 포함해 재현성 확보
  • 업로드 규칙 변경은 데이터 계약 버전 업데이트로 반영

CI/CD

  • 매우 작은 규칙 변경도 PR→CI→스테이징 자동화로 일괄 반영
  • 캐시된 예측 모델 버전과 matrix 버전을 별도 태그로 관리
  • 회귀 확인 항목으로 사용률 KPI와 실패율을 PR 게이트에 반영

Monitoring

  • 요청 처리 지연, 매칭 후보 수, 실패 사유를 KPI로 집계
  • score matrix 버전별 선택율을 추적해 정책 성능 편차를 점검
  • 에러/경고를 사용자/영업소 단위로 분리 집계해 대응 우선순위 정립
Troubleshooting / Operation

운영 이슈 정리

증상 원인 조치 결과
요청 지연이 간헐적으로 증가 후보 생성/필터 단계가 서로 다른 코드경로에서 중복 실행 API 레이어를 후보 생성-필터-채점으로 단계화하고 캐시 적용 피크 타임 안정화 및 대기열 증가 억제
점수 산정 이유를 추적하기 어려움 예측값, 규칙 값, 최종 스코어 로그 규격 분리 미흡 요청 trace id 기반 3단계 점수 로그로 구조화 장애 조치 시간이 짧아지고 규칙 검증이 쉬워짐
영업소 요구사항 반영이 지연 규칙 반영이 코드 수정 + 수동 배포에 의존 score matrix와 규칙을 운영 저장소에서 버전 관리 1주 이내 반영이 가능한 운영 루틴 확립
Results

결과

운영 안정성

규칙과 모델을 분리해 정책 변경 시 배차 응답 안정성이 유지됨

반영 속도

요구사항 반영 주기를 1주 내로 단축해 팀 협업 속도를 개선

성과

사용률 20% → 80%, 일부 영업소는 97%에 도달

유지보수성

예측 모델 변경과 정책 변경의 경로가 분리되어 위험도가 낮아짐

Lessons Learned

교훈

운영 중심 설계

정책이 자주 바뀌는 도메인은 모델 성능보다 계약과 반영 절차가 먼저 성패를 좌우합니다.

서비스 추적성

예측, 정책, 최종 결과를 분리 로그로 남기면 현업 이슈에 대한 설득력이 크게 올라갑니다.

확장 전략

동일 API에서 다중 규칙을 처리하기보다 matrix 기반으로 확장시키는 편이 운영 위험이 적습니다.
