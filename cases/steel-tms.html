Supporting Case

Steel TMS MLOps Backend

제품별 배차 가능 차량을 예측한 뒤, 영업소별 요구사항은 Scorer/Score Matrix로 추상화해 운영 가능한 배차 정책 백엔드로 정리한 사례입니다.

핵심 메시지

예측 + 규칙 + 운영 API를 한 번에 결합해 수동 조율 의존도를 낮췄습니다.

기간

2024.02 ~ 2024.11

주요 역할

멀티클래스 예측 파이프라인 적용, score matrix 설계, 배포/모니터링 고도화

기술

Python, FastAPI, pandas, Redis, GitLab CI/CD, Prometheus/Grafana

결과

사용률 20% → 80%, 일부 영업소 97%까지 개선 및 SLA 안정화

Context

도메인 문제를 서비스 제약으로 번역한 이유

각 영업소는 차량, 제품, 마감 시간 규칙이 다르게 동작해 모든 로직을 하드코딩으로 처리하기 어려웠습니다. 모델 예측값만으로는 현장 규칙 반영이 충분하지 않아, 운영자가 해석 가능한 점수 체계가 필요했습니다.

Backend Problem

처리 구조를 왜 바꿨는가

문제

  • 영업소별 규칙이 다르지만 API 경로가 하나의 흐름에 묶여 있었음
  • 예측 결과만으로는 배차 정책 우선순위가 불명확함
  • 재학습/배포/테스트 간 일관성이 떨어졌음

접근

  • multi-class 예측 출력을 정책 엔진이 이해할 수 있는 스키마로 정리
  • 영업소 특화 Scorer를 API로 분리해 규칙을 유지보수 가능하게 구성
  • CI/CD와 모니터링 지표를 API 응답 단계까지 통합
Architecture / Figure

Figure로 보는 핵심 설계

ML 예측값과 운영 계약

이 그림에서 봐야 할 점

  • 예측 출력은 직접 배차 판단이 아니라 scoring 입력으로 변환됩니다.
  • 확률값은 규칙 엔진에서 가중치 조합의 시작점 역할을 합니다.
  • 운영 팀 해석이 가능한 형태로 출력값의 의미를 유지합니다.

Scorer / Score Matrix

이 그림에서 봐야 할 점

  • 영업소별 정책은 scorer 단에서 추상화돼 API 로직은 고정됩니다.
  • 예측 확률과 제약 점수를 결합해 후보 순위를 산출합니다.
  • 점수 체계는 정책 변경 이력을 API 단에서 추적하기 쉽게 만듭니다.
Key Design Decisions (ADR)

ADR

ADR-01 / 모델-정책 분리

Decision
예측 API와 scorer API를 분리
Context
예측 정확도 개선만으로는 영업소별 정책을 반영하기 어려움
Why
정책 변경 주기와 모델 업데이트 주기를 독립화
Trade-off
계약 인터페이스 관리 비용이 함께 생김

ADR-02 / Score Matrix 구조

Decision
Scorer를 행렬형 score 모델로 통일
Context
영업소 규칙마다 분기 구조가 달라 유지보수가 증가
Why
규칙 조합을 일관된 형식으로 관리하기 위함
Trade-off
초기에는 규칙 이행 데이터 정합성 작업이 큼

ADR-03 / 운영 자동화

Decision
CI/CD와 모니터링을 배차 API 라이프사이클까지 정합
Context
데이터 변경 시 배포·검증·문서화가 분리되어 지연 발생
Why
서비스 안정성을 지키기 위해 검증 경로를 고정
Trade-off
파이프라인 규칙이 늘어나면 운영 관리 체계도 함께 성장
Implementation Details

구현 포인트

API Boundary

  • 예측 요청, scorer 요청, 정책 조회를 응답 스키마로 분리
  • 비정상 후보는 실패 코드와 사유를 명시해 상위 호출부 대응
  • ID 기반 재요청 정책을 둬 중복 연산을 줄임

Data Model

  • 예측 값, 사업소 정책, 점수 가중치를 각각 DTO로 분리
  • score matrix 계산 전/후 검증을 체크포인트로 둠
  • 점수 계산 과정을 로그로 저장해 감사 가능한 형태로 보존

CI/CD & Monitoring

  • PR 기반 테스트, 점수 스키마 변경 검증, 배포 승인으로 일관성 확보
  • 배차 지연률, 후보 수, 오류율 KPI를 API 단계까지 추적
  • 문제 재현 데이터를 쿼리 가능한 형태로 남김

운영 체계

  • 영업소별 Scorer 설정은 운영자 권한 기반으로 제어
  • 이상치 후보는 실시간 알림으로 분리 조회
  • 릴리즈마다 회귀 영향 범위를 표준 문서로 업데이트
Troubleshooting / Operation

운영 이슈와 개선

증상 원인 조치 결과
영업소 규칙 반영이 느림 정책 분기와 모델 API가 묶여 있음 Scorer를 별도 레이어로 분리하고 배포 경로 고정 규칙 변경 리드타임이 단축
후보 계산이 동일하게 안 나옴 데이터 정합성 체크가 부족 data contract 점검을 예측·점수 계산 사이에 추가 재현성 있는 디버깅이 가능해짐
모니터링 공백 성능 지표가 API 단계에서 끊김 score matrix 전후 KPI를 동일 trace로 수집 장애 탐지와 회귀 대응 속도 향상
Results

결과

성능

요건별 배차 처리율이 개선되며, 전체 사용률이 20%에서 80~97%로 상향

구조적 안정성

영업소 특화 규칙을 score matrix로 분리해 코드 변경 범위를 축소

운영 신뢰성

예측, 규칙, 응답 단계까지 추적 가능한 로그 체계를 확보

Lessons Learned

교훈

도메인 + 운영 합의

ML 점수의 정확도만으로는 충분하지 않고 운영 해석성이 병행되어야 합니다.

계약 설계

데이터 계약을 앞서 정하면 정책/학습/출력 단계에서 오류 전파가 줄어듭니다.

SLA 친화 설계

모델 점수와 규칙을 한 레이어에서 묶으면 응답 품질과 성능 조정이 쉬워집니다.
