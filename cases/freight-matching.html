Supporting Case

Freight Matching API Optimization

복화 후보 탐색을 단순 loop 기반이 아니라 후보쌍 생성 + 벡터화 필터 파이프라인으로 바꿔, B2C 모바일 3초 응답 요구를 만족한 API 최적화 작업입니다.

핵심 메시지

기존 배차 기능 재사용 구조에서 API 경계를 정리하고, 검색 후보 생성/필터링을 대규모 벡터 연산으로 재구성했습니다.

기간

2024.06 ~ 2024.12

역할

기존 기능 분해 설계, API 분리, 데이터 파이프라인 설계, 성능 실험

기술 스택

Python, FastAPI, pandas, Redis, PostgreSQL, CI/CD, Prometheus

핵심 성과

처리 시간 900초 → 1초, 모바일 응답 지연 구간을 제거

Context

왜 이 문제가 중요했는가

B2C 모바일 서비스는 응답 체감 속도가 핵심이었고, 3초를 넘는 후보 탐색은 실제 사용성에서 곧바로 이탈로 이어졌습니다. 기존에는 오래된 배차 로직을 재사용하면서도 후보군 탐색이 증가할수록 응답이 급증했습니다.

Backend Problem

왜 기존 구조로는 안 됐는가

병목의 본질

  • 기존 배차 기능이 monolith 경로에 묶여 확장 지점이 불분명
  • 재사용된 submodule은 배차 특화 로직 변경에 취약했음
  • 후보군 생성과 정책 필터가 섞여 있어 성능 병목 분석이 어려웠음
  • 캐시/재시도 정책이 없어 장애 시 회복 경로가 느림

해결 방향

  • 기존 배차 기능을 API boundary로 분리해 책임을 분해
  • 복화 후보를 self-merge로 대량 생성 후 boolean indexing으로 규칙 필터
  • 성능 계측 지점을 공통 로그에 고정해 병목을 정량적으로 추적
  • 요청 캐시와 임계치 정책으로 피크 대응력을 확보
Architecture / Figure

Figure로 보는 핵심 구조

기존 기능 분리 구조

이 그림에서 봐야 할 점

  • 기존 모듈 재사용의 장점을 살리되 API 경계를 분리해 영향도를 고정
  • 후보 생성, 점수 산정, 응답 조합을 분리해 확장성 확보
  • 개별 구간의 SLA 추적이 가능해졌음

Self-merge + Boolean indexing 최적화

이 그림에서 봐야 할 점

  • 후보쌍을 벡터화 연산으로 생성해 반복 루프를 제거
  • 복합 규칙은 순차 loop가 아니라 정렬된 조건식으로 필터링
  • 결과적으로 900초급 지연을 1초대로 축소
Key Design Decisions (ADR)

ADR 카드

ADR-01

Decision
기존 배차 도메인을 API 레이어 단위로 재분해
Context
기능이 너무 많은 경로를 공유해 성능 병목의 원인을 추적하기 어려웠음
Why
문제 영역별 SLA를 분리하고 운영 이슈를 빠르게 고립하기 위함
Trade-off
초기에 계약 스키마 재정의와 연동 조율 비용 발생

ADR-02

Decision
후보 생성 방식을 self-merge + boolean indexing으로 전환
Context
loop 기반 탐색은 수량 증가 시 곱셈적으로 시간이 증가
Why
pandas vectorization의 캐시/메모리 특성으로 대규모 후보군을 안정 처리하기 위함
Trade-off
조건식 관리가 명확하지 않으면 디버깅 비용이 커질 수 있음

ADR-03

Decision
성능 KPI를 API별로 계측하고 timeout·fallback 정책 고정
Context
병목 탐지 지점이 섞여 있어 원인 분석이 늦어졌음
Why
운영 단계에서 3초 제한 위반을 즉시 감지하고 대응하기 위함
Trade-off
모니터링 항목 유지관리와 경보 튜닝이 필요
Implementation Details

구현 포인트

API Boundary

  • 매칭 후보 생성 API, 규칙 필터 API, 응답 조합 API로 단계 분리
  • 각 API는 입력 검증과 idempotency를 명시적으로 적용
  • 실패 응답에는 재시도 가능 구간과 오류 분류 코드를 함께 전달

Queue / Runtime

  • 대용량 후보 생성 구간은 작업 큐 우선 처리로 장기 점유를 완화
  • 요청 우선순위는 피크 분산 전략에 따라 가중치 부여
  • 재시도는 지수 백오프와 재요청 토큰으로 제어

Data Contract

  • 자유 형식 객체 대신 표준 필드 기반 DTO 채택
  • 후보군 생성 조건과 필터 조건을 별도 스키마로 관리
  • 버전 충돌을 방지하기 위해 요청/응답 스냅샷 저장

Monitoring / CI-CD

  • 시간별 p95 지연, 후보군 수, 필터 탈락률을 대시보드로 수집
  • 변경 전·후 성능 비교를 PR 체크로 자동 집계
  • 회귀 시 즉시 롤백 가능한 배포 정책 적용
Troubleshooting / Operation

운영 이슈 정리

증상 원인 조치 결과
특정 시간대 후보 탐색 지연 급증 요청량 증가 시 loop 탐색 구간이 선형 이상으로 확대 self-merge 전처리와 boolean 필터 조합으로 처리 파이프라인 교체 평균 응답이 대폭 개선되고 지연 변동성이 줄어듦
모듈 재사용 후 정책 누락 공유된 submodule에 프로젝트별 제약이 섞임 기능을 API 경계에서 캡슐화하고 계약 검증을 추가 요청 간섭이 줄고 운영 품질이 안정화됨
3초 SLA 위반 알림 빈번 성능 지표가 API 단위로 나뉘지 않아 알람이 모호 후보 생성/필터/결합 구간을 별도 지표로 분리 원인 구간을 즉시 식별해 조치 속도 증가

결과

핵심 성능 지표

Lessons Learned

교훈

문제 재정의의 중요성

조합 탐색 문제를 데이터 파이프라인 문제로 재정의하면 최적화 폭이 훨씬 커집니다.

API 단위 운영성

기능 분할은 성능뿐 아니라, 추적성과 장애 대응성도 함께 개선합니다.

지표의 구조화

지표를 단계별로 분리하면 제품 팀과 운영 팀 모두 같은 언어로 문제를 협업할 수 있습니다.
