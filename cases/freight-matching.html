Supporting Case

Freight Matching API Optimization

기존 조합 탐색 loop를 API 재설계로 재정의해, B2C 서비스 3초 제약을 맞추기 위해 후보 탐색을 900초에서 1초로 단축했습니다.

핵심 메시지

문제를 반복문에서 벡터화 문제로 재정의하면 운영 성능을 크게 개선할 수 있습니다.

기간

2024.06 ~ 2024.12

주요 역할

기존 기능 API 분리, 후보 생성 구조 개선, 성능 검증

기술

Python, FastAPI, pandas, Redis, PostgreSQL, Prometheus

결과

응답 900초 → 1초, 모바일 서비스 3초 이내 목표 대응

Context

왜 구조를 바꿨는가

B2C API는 응답 지연이 직접 사용자 경험으로 이어집니다. 기존 방식은 후보 생성이 브루트포스에 가까워 수십~수백 건 조합 탐색에서 과도한 시간이 소모되었고, 서비스 제약을 맞추기 어렵습니다.

Backend Problem

일반 구조의 한계

문제

  • 기존 모듈 결합이 느슨해 API 경계가 모호
  • Submodule 재사용은 분할은 됐지만 병목은 남음
  • 루프 기반 후보 생성이 병렬화되어도 지표 개선이 제한적

접근

  • API boundary를 분리해 후보 생성 책임을 명확히 정리
  • pandas self-merge로 후보쌍 생성 후 boolean indexing으로 필터링
  • 후보 쌍 단계에서 규칙을 앞당겨 계산량을 줄임
Architecture / Figure

도식으로 보는 성능 전략

기능 API 재사용 구조

이 그림에서 봐야 할 점

  • 기존 기능을 그대로 재사용하면서 병목 구간을 API로 명확 분리했습니다.
  • 요청의 각 단계가 trace 가능해져 운영 대응이 쉬워집니다.
  • 확장 시 모듈 교체 범위가 줄어듭니다.

Self-merge + Boolean indexing

이 그림에서 봐야 할 점

  • 모든 조합을 직접 for loop로 순회하지 않고 벡터화
  • 조건 기반 필터링을 한 번에 적용해 후보 수를 급감
  • 실시간 응답 제약에 맞는 처리량을 확보
Key Design Decisions (ADR)

ADR

ADR-01 / API Boundary 재정의

Decision
기존 배차 모듈을 재사용 가능한 API로 경계 분리
Context
루프 중심 구현은 API 확장성과 배포 분리가 어려움
Why
기능 단위를 독립 배포 가능하게 하여 성능 최적화를 반복 가능
Trade-off
초기 설계에서 인터페이스 버전 정책이 필요

ADR-02 / Self-merge 채택

Decision
후보쌍 생성과 필터링을 vectorized 연산으로 변경
Context
고차원 조합에서 루프 비용이 지연의 주 원인
Why
계산량을 연산 모델 수준에서 줄여 SLA 대응 속도 확보
Trade-off
쿼리 식과 필터 규칙의 가독성 관리가 필요

ADR-03 / 운영 측면 강화

Decision
캐시 키/요청 이력/fallback 메시지를 함께 관리
Context
고속 응답 조건에서 장애 포인트 추적이 어려움
Why
성능 개선 후에도 장애 대응력을 유지하기 위해
Trade-off
추가로 저장 항목이 늘어 운영 비용이 증가
Implementation Details

구현 포인트

API Boundary

  • 요청 API, 후보 생성 API, 점수 평가 API를 분리
  • 요청 키 기반 idempotency를 적용해 중복 요청 통제
  • 에러 응답 스키마를 통일해 상위 호출부 처리 일관화

Queue / Runtime

  • 비동기 처리와 동기 응답 경로를 분리
  • 캐시를 통한 반복 계산 회피
  • 임시 fallback 경로를 두어 타임아웃 대응

Data Contract / Filtering

  • 필터 규칙을 DataFrame 단계에서 명시화
  • 조건별 boolean mask를 중앙에서 관리
  • 후보쌍 규칙 변경 시 영향 범위를 최소화

Monitoring

  • p95 응답시간과 실패율을 지속 모니터링
  • 배포 전/후 성능 비교를 PR 단위로 기록
  • 임계치 초과 시 alert를 바로 Slack runbook로 연결
Troubleshooting / Operation

운영 이슈와 조치

증상 원인 조치 결과
초기 응답이 수 분 이상 소요 조합 탐색이 중첩 루프로 구성 self-merge + boolean indexing로 후보 생성 구조 변경 응답 900초에서 1초로 단축
모듈 재사용 후 병목 지속 규칙 적용 위치가 API 후반부에 집중 필터 규칙을 앞단에서 적용 후보 수 감소로 전체 처리량 증가
3초 응답 제한 충족 실패 fallback/재시도 경로 미정의 타임아웃 기반 응답 경로와 캐시 전략 정비 모바일 SLA 충족률이 향상

Lessons Learned

교훈

문제 재정의

연산 비용은 알고리즘 교체보다, 경계 재설계에서 더 크게 개선되는 경우가 많습니다.

API 계약

계약 경계가 명확할수록 성능 개선의 실효성과 팀 협업 효율이 함께 올라갑니다.

운영 신뢰성

속도 개선은 모니터링 없는 개선이 아니라, 모니터링이 동반될 때만 지속됩니다.
