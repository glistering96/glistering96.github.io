Main Case

자동차선 적부 최적화 Backend Service

복잡한 선박 적재 도메인을 계산 가능한 graph/data model로 추상화하고, 5~20분 걸리던 장시간 AI 추론을 기존 시스템과 연결 가능한 운영형 backend로 만들었습니다.

기간

2024.01 ~ 2026.03

역할

도메인 분석, API/Queue/상태 설계, 장애 조치, 배포 운영

기술 스택

Python, FastAPI, RabbitMQ, Redis, Kubernetes, Istio, Helm

핵심 성과

추론 지연 5~20분 → 2~3분, scale-out 상태 추적 안정화

운영 포인트

API boundary 분리 + callback 라우팅 + 중앙 상태/재처리 정책

Context

왜 이 문제를 먼저 풀었는가

적부 최적화는 높이, 공간, 통행 동선, 선적 순서를 모두 고려해야 하는 구조적 문제였습니다. 추론이 길어질수록 동기 호출은 연결 유지 비용이 커졌고, 장애 발생 시 복구 위치를 찾기 어려웠습니다.

Backend Problem

일반적인 API 구조만으로는 운영 목표를 만족 못했다

상태

  • 요청별 상태를 API pod 내부 메모리에서만 관리하면 scale-out에서 경합이 생김.
  • 요청 상태, 재시도 카운트, callback 결과를 통합 조회하기 어려웠습니다.
  • 비정상 종료 시 유실된 처리건을 즉시 추적하지 못했습니다.

장시간 작업/동시성

  • 동기 호출은 timeout 위험과 자원 점유를 유도해 처리량을 떨어뜨렸습니다.
  • callback 응답이 원 요청 pod로 돌아오지 않을 수 있는 구조였습니다.
  • 모델 출력 포맷 차이로 계약 불일치가 반복 발생했습니다.
Architecture / Figure

현업 도메인과 시스템 경계가 만나는 지점

자동차선 내부/도메인 이해

이 그림에서 봐야 할 점

  • 작은 차이의 통행 제약이 최종 적재 가능성을 크게 바꿉니다.
  • 도메인 규칙이 입력 모델에 반영되어야 API가 안정적으로 동작합니다.
  • 그림에서 보듯이 적재 대상은 선형 적치가 아니라 경로 제약 기반 객체입니다.

선박 공간의 graph 추상화

이 그림에서 봐야 할 점

  • 구역을 node로, 인접 이동을 edge로 모델링해 제약을 정규화했습니다.
  • 선박 유형별 규칙은 edge 가중치/제한으로 처리할 수 있습니다.
  • 새 선박 유형 추가 시 신규 규칙은 데이터만 추가하는 방향으로 정리했습니다.

4층 / 5층 / 6층 통행 예시

이 그림에서 봐야 할 점

  • 층별로 통행 가능 구간이 달라 같은 차량도 적재 결과가 달라집니다.
  • 층간 이동 제약을 계산에 반영해야 실제 현장 규칙과 일치합니다.
  • API 입력에서 층별 제약을 분리 처리해 오류를 줄였습니다.

Hold, Deck, Vessel, Cargo 데이터 모델

이 그림에서 봐야 할 점

  • 도메인 객체를 분리해 계산/변환 로직을 책임별로 나눴습니다.
  • 시스템별 입력 차이는 변환 계층에서 처리했습니다.
  • 모델 교체 시 도메인 중심 모델은 변경 범위를 최소화합니다.

비동기 callback 구조 (요청-응답 분리)

이 그림에서 봐야 할 점

  • 요청 API는 결과 대기를 하지 않고 queue로 흐름을 넘깁니다.
  • 모델 완료 callback은 별도 endpoint로 받아 상태를 갱신합니다.
  • 요청-완료의 분리는 timeout와 자원 점유를 줄였습니다.

Scale-out AS-IS 문제

이 그림에서 봐야 할 점

  • 동일 요청이 다른 pod에서 callback을 받으면 로컬 상태를 찾지 못합니다.
  • 장애 복구 시 어느 단계에서 중단됐는지 추적이 어렵습니다.
  • scale-out가 늘수록 비결정성 문제가 증폭되었습니다.

Scale-out TO-BE 구조

이 그림에서 봐야 할 점

  • 요청/결과를 조회 가능한 공용 상태 저장소로 이동했습니다.
  • consumer 멱등성 정책으로 중복 처리 리스크를 줄였습니다.
  • 노드 장애 시에도 재처리 경로가 분명해집니다.

표준 데이터 모델 계약

이 그림에서 봐야 할 점

  • 입출력 형식 차이는 표준 모델 기준으로 한 번만 변환합니다.
  • 모델 변경 시 외부 연동은 adapter에서만 조정합니다.
  • 운영에서는 계약 검증 실패를 사전 방지할 수 있습니다.

Lessons: 계약 정합성과 직렬화

이 그림에서 봐야 할 점

  • Pydantic으로 입력/출력 계약을 선행 검증해 런타임 오류를 크게 줄였습니다.
  • pickle 기반 전송은 버전 의존성이 컸고, JSON 계약 전환으로 안정화했습니다.
  • consumer 동시성은 lock/token 정책으로 재처리 규칙을 통제했습니다.
Key Design Decisions (ADR)

아키텍처 선택 기준

ADR-01 / Long job 처리

Decision
요청 API와 추론 엔진 실행을 분리하고 callback으로 완료 신호를 수신
Context
동기 API에서 수분 단위의 연결 유지 비용이 계속 증가
Why
장시간 작업을 요청 생명주기와 분리해 안정적 처리량 확보
Trade-off
요청 추적용 상태 저장과 재시도 관리가 추가로 필요

ADR-02 / 상태 일관성

Decision
pod-local 상태를 제거하고 Redis 기반 state store 사용
Context
scale-out 환경에서 callback이 원 요청 pod와 불일치
Why
어떤 노드에서든 동일 요청을 조회할 수 있어야 처리 성공률이 올라감
Trade-off
Redis 가용성·TTL 정책 운영 책임이 생김

ADR-03 / 데이터 계약

Decision
사내 모델/시스템은 내부 표준 모델 중심으로 변환
Context
외부 입력 형식이 자주 달라 통합 비용이 과다
Why
도메인 불변 조건을 계약으로 고정해 장기 운영 비용 절감
Trade-off
초기 설계 단계에서 변환 레이어 구현 비용 존재
Implementation Details

API boundary, queue, runtime, CI/CD, monitoring

API Boundary

  • 요청 API: 인증, 스키마 검증, idempotency key 발급
  • 결과 API: callback payload 유효성 검사 및 상태 전이
  • 운영 조회 API: 요청 이력, stage, 재시도 횟수 공개

Queue / Callback

  • 요청 접수는 producer로 enqueue
  • consumer에서 재시도 횟수·보류 상태를 관리
  • callback은 요청 id 기준으로 처리 후 상태 동기화

Data Model

  • Pydantic 기반 입출력 검증을 사전 적용
  • 표준 데이터 계약으로 도메인/엔진/모델 변환 계층 분리
  • JSON 직렬화를 기본으로 사용해 버전 이슈를 완화

Runtime / CI-CD / Monitoring

  • Helm chart와 values로 환경별 배포를 템플릿화
  • Istio route를 활용해 모델 pod 분산을 제어
  • Trace ID 기반 로깅으로 처리 구간별 병목을 추적
Troubleshooting / Operation

운영 중 이슈 분석

증상 원인 조치 결과
요청이 쌓여도 처리율이 올라가지 않음 동기 처리 경로가 부분적으로 남아 있었음 요청·완료 흐름을 callback 중심으로 분리 평균 응답 연동 안정성이 개선됨
callback 결과를 못 찾는 현상 pod-local 상태 의존 구조 Redis 기반 전역 상태 조회로 변경 누락 및 재시도 실패 케이스가 감소
호환성 직렬화 이슈 pickle 포맷 의존과 라이브러리 버전 차이 JSON 계약 + encoder 표준화 배포간 데이터 파이프라인 호환성 개선
Results

결과

지연 단축

장시간 추론 파이프라인을 운영 가능한 비동기 API로 전환해 체감 지연을 5~20분 구간에서 2~3분대로 개선했습니다.

운영 안정성

scale-out 시에도 요청 상태와 callback이 정합적으로 연결되어 이탈 요청이 줄었습니다.

확장성

표준 데이터 모델을 기준으로 모델/엔진 교체 시 영향 범위를 최소화했습니다.

Lessons Learned

배운 점

설계 기준

계산 성능보다 먼저 상태 추적과 실패 복구를 정의해야 운영에서 살아남습니다.

운영 가독성

계약 검증 위치를 고정하면 장애를 발견하는 데 드는 시간이 현저히 줄어듭니다.

스케일 전제

단일 pod 기준 설계는 scale-out에서 깨지므로 처음부터 분산 상태 모델로 시작해야 합니다.
