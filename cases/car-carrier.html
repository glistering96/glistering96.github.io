Main Case

자동차선 적부 최적화 Backend Service

복잡한 선박 적재 도메인을 계산 가능한 graph/data model로 추상화하고, 5~20분 걸리던 장시간 AI 추론을 요청-응답이 단절되지 않는 운영 가능한 backend로 만들었습니다.

기간

2024.01 ~ 2026.03

역할

기획/아키텍처 설계, 백엔드 구현, 운영 이력 관리, 장애 대응

Stack

Python, FastAPI, Redis, RabbitMQ, Kubernetes, Istio, Helm

핵심 성과

추론 처리 시간 5~20분 → 2~3분, scale-out 시 동시 처리 안정성 확보

운영 포인트

queue + callback, 상태 재처리, 관측성 로그, 모델 인터페이스 표준화

Context

왜 이 문제가 필요했는가

자동차선 선적은 단순히 빈칸을 채우는 문제가 아닙니다. 한 항차 안에서 선박의 높이, 선적지, 양하지, 차량 종류별 통행 경로가 서로 얽히기 때문에 선적 순서가 한 번만 틀려도 하역/출발이 지연됩니다. 또한 추론이 오래 걸리는 모델(5~20분) 특성상 기존 동기 API로는 호출 지연과 연결 유지 비용이 커서 실서비스 반영이 불가능했습니다.

Backend Problem

왜 일반 CRUD/API만으로는 안 되었는가

문제의 본질

  • 동기형 요청은 장시간 연결 유지가 필요해 API 인스턴스 자원이 고정적으로 점유됩니다.
  • 모델 처리 시간이 요청 난도별로 크게 달라 타임아웃·재시도 정책이 어렵습니다.
  • scale-out 환경에서 pod 상태를 로컬에 두면 callback이 다른 인스턴스로 도착해 상태 추적이 깨집니다.
  • 도메인 데이터 형식이 모델, 엔진, 기존 사내 시스템마다 달라 계약 불일치가 발생합니다.

해결 방향

  • 요청 접수/검증/상태 기록/콜백 응답을 담당하는 API 레이어를 분리
  • 장기 실행은 queue로 위임하고 완료 후 callback으로 최종 응답 전달
  • 공용 상태(진행 단계, 재시도 카운트, callback 히스토리)를 Redis로 관리
  • 사내 시스템-엔진-외부 모델 간 인터페이스를 내부 표준 모델로 통일

구조를 기준으로 본 설계

자동차선 내부/도메인 이해

이 그림에서 봐야 할 점

  • 공간은 단순 2차원 면적이 아니라 통행 순서가 포함된 제약 그래프입니다.
  • 차량 적재는 높이/넓이 제약 뿐 아니라 작업 동선을 고려해야 합니다.
  • API로 넘기는 입력값은 이 제약을 반영해야 정확도가 높습니다.

선박 공간 Graph 추상화

이 그림에서 봐야 할 점

  • 적재 가능한 구역을 node, 연결을 edge로 모델링
  • 선박 타입과 층별 제약을 동일 패턴으로 관리
  • 추가 선박 유형 확장 시 데이터 구조 변경 비용을 낮춤

4/5/6층 통행 예시

이 그림에서 봐야 할 점

  • 층마다 이용 가능한 루트와 진입 포인트가 다릅니다.
  • 높은 층일수록 통행 제약이 강해지는 조건을 반영했습니다.
  • 같은 차량군도 층별 경로가 다르면 배치 결과가 달라집니다.

Hold / Deck / Vessel / Cargo 모델

이 그림에서 봐야 할 점

  • 도메인 주체를 명시적으로 분리해 계산 단계 입력을 단순화
  • 시스템별 포맷 차이는 엔진 호출 전후의 변환점에서만 처리
  • 새 규격 모델 추가 시 데이터 모델만 확장하면 됨

비동기 callback 구조

이 그림에서 봐야 할 점

  • 요청 API는 처리 완료 대기를 하지 않고 queue에 위임
  • 모델 완료 후 callback을 별도 endpoint로 받아 상태 업데이트
  • 폴링 기반 흐름 대비 자원 낭비를 크게 줄임

Scale-out AS-IS

이 그림에서 봐야 할 점

  • 요청 처리 pod와 callback 처리 pod가 분리될 가능성
  • pod-local 상태로는 scale-out에서 완전 보장 불가
  • 요청 유실 및 중복 처리 위험이 존재

Scale-out TO-BE

이 그림에서 봐야 할 점

  • 상태를 공용 스토어에 두어 어디서 들어온 callback도 동일 요청을 찾음
  • consumer 재처리 및 멱등성 키로 중복을 제어
  • 장애 복구 시 장애 지점부터 재개 가능

표준 데이터 계약

이 그림에서 봐야 할 점

  • API/엔진/모델 사이에 내부 계약 모델을 둠
  • 형식 변환을 이 레이어에서 일괄 처리
  • 새 모델 교체 시 하부 조합만 변경하면 됨

Lessons: 계약 정합성과 직렬화

이 그림에서 봐야 할 점

  • 요청/응답 스키마는 운영 전 단계에서 검증해야 런타임 이슈가 줄어듭니다.
  • pickle 기반 전달은 버전 의존성이 커 JSON 계약으로 전환해 호환성을 확보했습니다.
  • consumer 동시성 제어는 멱등성 키와 상태 잠금으로 안정적으로 통합했습니다.
Key Design Decisions (ADR)

ADR 카드

ADR-01

Decision
요청 처리와 모형 추론을 분리하고 callback API로 완료 신호를 수신
Context
요청 지연이 긴 추론을 웹 요청 스레드에서 직접 처리하면 인스턴스가 장기 점유됨
Why
폴링 비용을 줄이고 SLA를 안정화하기 위해
Trade-off
callback 상태 추적 시스템이 추가됨

ADR-02

Decision
pod-local state 대신 Redis 기반 state store로 전환
Context
scale-out에서 callback이 원 요청 pod로 돌아오지 않는 현상 발생
Why
요청 연속성을 유지하고 재처리 가능하게 하려는 목적
Trade-off
Redis 운영 안정성(TTL, 장애 전환 정책)이 중요해짐

ADR-03

Decision
내부 표준 모델을 기준으로 외부 시스템/모델 변환 레이어 구현
Context
요청 형식이 시스템별로 달라 계약 불일치가 잦았음
Why
도메인 변경이 생겨도 핵심 서비스가 흔들리지 않게 하기 위해
Trade-off
초기 개발 초기 비용은 올라가나 장기 유지보수 비용이 낮아짐
Implementation Details

구현 포인트

API Boundary

  • 요청 API: 인증, 입력 검증, idempotency key 발급
  • 결과 API: callback payload 검증, 상태 갱신, 완료 알림 생성
  • 배치 API: 같은 선박의 동시 요청을 제한하거나 큐 우선순위 부여

Queue / Callback

  • 요청은 producer에서 queue에 저장 후 워커가 처리
  • callback은 결과 검증 후 상태 테이블 갱신 후 downstream 전달
  • 재시도는 exponential backoff와 멱등성 키로 제어

Data Model

  • Pydantic 기반 유효성 검사로 잘못된 입력 차단
  • 내부 계약 모델에서 모델/엔진 요청/응답을 상호 변환
  • binary pickle 대신 JSON 직렬화 경로로 마이그레이션

Runtime / CI & Monitoring

  • Kubernetes + Istio로 pod 상태/라우팅 정책 분리
  • 헬름 values 파일로 환경별 배포를 템플릿화
  • 요청 ID 전파 기반으로 로그/트레이싱의 상관관계 분석
Troubleshooting / Operation

운영 이슈와 조치

증상 원인 조치 결과
장시간 요청이 들어오면 API 인스턴스 메모리가 고정 점유 동기 처리에서 추론 API를 그대로 기다림 요청 접수 후 queue 위임, callback 분리 서버 부하가 완화되고 처리량이 안정화됨
callback이 요청 pod가 아닌 다른 pod로 도착 scale-out 이후 pod-local 상태 사용 Redis 상태 키 기반 조회로 일원화 누락/지연이 크게 줄고 추적이 쉬워짐
직렬화 호환성 오류로 특정 데이터가 깨짐 pickle 기반 전달 + 라이브러리 버전 차이 Pydantic JSON 직렬화와 전용 커스텀 encoder 적용 호환성 이슈와 장애 재현 시간이 줄어듦
재처리 시 중복 실행 위험 consumer 동시성 제어 부재 consumer id, token, 상태 마이크로락 정책 적용 중복 계산이 감소하고 결과 정합성 개선
Results

결과

추론 지연 단축

5~20분이 걸리던 장시간 추론 처리 플로우를 2~3분 수준으로 개선

운영 안정성

scale-out 콜백 추적 실패를 Redis 중심 상태로 수렴해 장애 대응이 단순화됨

교체 유연성

모델과 엔진이 바뀌어도 내부 계약만 유지하면 인터페이스 파급을 줄일 수 있음

운영 협업

API 경계와 로그 규칙을 고정해 운영 팀과 개발팀이 동일 뷰로 문제를 진단

Lessons Learned

교훈

설계 우선순위

알고리즘은 성능 개선에 필요하지만, 운영에서는 상태 설계와 API 계약이 핵심이라는 점을 재확인했습니다.

검증 체계

모델 추론 이전·이후의 validation을 먼저 설계하면 장애 전파를 크게 줄일 수 있었습니다.

확장성

scale-out 환경은 분산 상태를 가정한 설계를 요구하며, 이 점이 프로젝트 전체의 안정성 기준이었습니다.
